# AI Story Lab 开发约定

## 项目目标

本项目用于构建一个可演示、可验证的 AI 互动故事 MVP：用户在免登录手机网页中填写四项故事设定，由可替换的服务端生成器产出经过 Zod 校验的分支故事，通过两次二选一到达结局，并可刷新恢复或重新创作。默认使用 Mock，显式配置后可使用 OpenAI。

产品定义、范围和验收标准以 `docs/PRODUCT.md` 为准；AI 输出质量以 `docs/EVALUATION.md` 为准。

## 目录结构

项目初始化后按以下职责组织目录：

```text
.
├── app/            # Next.js 路由、页面、布局和服务端接口
├── components/     # 可复用 UI 与故事交互组件
├── lib/            # 故事状态、AI 调用、Zod schema、事件统计等领域逻辑
├── content/        # 版本控制内的静态故事配置
├── public/         # 预制图片、字体等静态资源
├── tests/          # 跨模块及端到端之外的集中测试资产（如需要）
├── docs/           # 产品定义、决策记录和 AI 质量评测
└── AGENTS.md       # 本文件，项目级开发约束
```

优先将测试与被测模块放在相邻位置；只有共享夹具或跨模块测试才放入 `tests/`。不要仅为匹配目录示意而创建空目录。

## 固定技术栈

- Next.js
- TypeScript
- Tailwind CSS
- Zod：所有外部输入、AI 结构化输出和配置边界的运行时校验
- Vitest：单元测试和集成测试

未经重要产品或技术决策记录，不得替换上述核心技术栈。

## 修改与验证要求

每次修改后必须依次运行：

1. `lint`
2. `typecheck`
3. `test`

使用项目 `package.json` 中对应的脚本，例如 `npm run lint`、`npm run typecheck`、`npm run test`。如果工程尚未初始化或脚本尚未配置，必须在交付说明中明确指出哪些检查未运行及原因；不得声称不存在的检查已经通过。

修复检查失败时只处理与当前任务相关的问题，不得借机重写无关代码或扩大范围。

## MVP 边界

不得自行增加以下能力：

- 登录、账户体系或用户身份绑定
- 自建数据库或跨会话用户状态持久化
- 支付、订阅、虚拟货币或其他商业化能力
- 多故事、多题材或多章节
- 自由聊天、无限续写或复杂分支树
- OpenAI 以外的模型供应商、多供应商路由或浏览器端模型调用
- 运行时实时图片或视频生成
- 社区、排行榜、好友、积分等社交或游戏化系统

预制静态视觉素材属于 MVP 范围；运行时实时图片生成不属于 MVP 范围。轻量事件统计可以接入，但不得以此为理由引入业务数据库。

如任务与 `docs/PRODUCT.md` 冲突或需要扩大上述边界，先停止实现并请求产品确认。

## 产品决策维护

涉及以下内容的重要决定必须同步更新 `docs/DECISIONS.md`：

- MVP 范围或非目标
- 用户流程和关键交互
- 技术栈、外部服务或数据持久化方式
- AI 生成策略、失败兜底或内容安全策略
- 指标定义、事件统计或评测通过标准

决策记录必须包含时间、决定、原因、替代方案和后续验证方式。不要覆盖历史决定；变更决定时新增记录并说明关联关系。

## 实现原则

- 以完成一条可测试的端到端用户路径为优先，不擅自扩大 MVP。
- 所有 AI 结构化输出必须通过 Zod 校验；失败时不得进入游玩流程，并按 `docs/PRODUCT.md` 返回可理解提示。Mock 模式是开发、测试和运维兜底，不得把 Mock 静默伪装成 AI 输出。
- 模型调用只能出现在服务端，API Key 只能从服务端环境变量读取，不得写入客户端代码、响应、LocalStorage 或日志。
- 实现必须满足 `docs/PRODUCT.md` 的验收标准。
- AI 输出上线前必须按 `docs/EVALUATION.md` 评测，任一维度低于 3 分的案例不得视为通过。
- 不记录可直接识别个人身份的信息。
