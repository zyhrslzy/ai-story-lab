# 产品决策记录

本文件记录会影响产品范围、用户流程、技术取舍或验证方式的重要决定。新决策按时间追加，不覆盖旧记录；若决定发生变化，应新增记录并关联原决定。

## 记录格式

### YYYY-MM-DD：决策标题

- **时间**：YYYY-MM-DD
- **决定**：最终采用的方案。
- **原因**：当前阶段采用该方案的依据。
- **替代方案**：讨论过但未采用的方案及未采用原因。
- **后续验证方式**：需要观察的数据、用户反馈，以及重新评估此决定的触发条件。

## 初始决策

### 2026-07-18：第一版不做登录

- **时间**：2026-07-18
- **决定**：第一版使用免登录体验，以匿名会话标识和会话级状态保存当次选择，不建设账户体系。
- **原因**：产品需要验证的是三分钟故事能否产生情绪回报和分享意愿。登录会增加进入阻力，并将开发时间转移到身份、授权和账户管理上，不能直接降低当前核心风险。
- **替代方案**：手机号验证码登录、第三方社交账号登录、游客体验后再登录。前两种会增加开发与合规成本；“先游客后登录”仍需要账户和状态合并能力，第一版没有必要。
- **后续验证方式**：观察完成后是否有用户持续提出保存历史结局、跨设备同步或找回内容的需求。当首轮用户中至少 30% 明确需要跨会话保存，或产品开始提供多次回访价值时，重新评估登录。

### 2026-07-18：第一版不做数据库

- **时间**：2026-07-18
- **决定**：故事内容使用版本控制内的静态配置；当次选择保存在客户端会话状态中；行为验证使用轻量事件统计，不自建业务数据库。
- **原因**：第一版只有一个故事，不需要动态内容管理、用户数据持久化或复杂查询。数据库会引入数据模型、迁移、部署和运维工作，但不会直接提升核心体验。
- **替代方案**：自建关系型数据库、托管数据库、无服务器键值存储。它们都能支持持久化，但现阶段没有必须持久化的业务对象；轻量事件统计可满足初始指标验证。
- **后续验证方式**：当需要运营人员动态发布多个故事、保存用户历史、支持跨会话状态，或轻量事件统计无法回答关键产品问题时，重新评估数据库和数据模型。

### 2026-07-18：第一版不做运行时实时图片生成

- **时间**：2026-07-18
- **决定**：第一版使用预先制作的静态视觉素材和固定结局卡模板，只实时生成结构化文本结局。
- **原因**：实时图片生成会增加等待时间、成本、风格漂移、内容安全和失败处理复杂度。第一版需要先验证互动故事和专属结局是否值得分享，而不是验证图像生成能力。
- **替代方案**：每个剧情节点实时生成图片、仅结局卡实时生成图片、后台异步生成图片。三者都会增加体验不确定性；预制素材更有利于控制氛围和稳定演示质量。
- **后续验证方式**：访谈用户是否认为固定视觉削弱了“专属感”，并比较结局文案与视觉对分享意愿的影响。当稳定文本版本达到初始完成率和分享意图率目标，且用户明确期待个性化视觉时，再进行小流量图片生成实验。

### 2026-07-18：第一版不做支付

- **时间**：2026-07-18
- **决定**：第一版完全免费，不加入付费入口、商品、订阅或虚拟货币。
- **原因**：当前尚未验证用户是否愿意完成、分享或重玩。过早加入支付会混淆核心价值验证，并增加订单、退款、风控和合规成本。
- **替代方案**：单故事付费、订阅、付费解锁额外结局、购买生成次数。这些模式都依赖稳定的留存或内容需求，而第一版没有足够证据支持。
- **后续验证方式**：先验证完成率、分享意图率和重玩率。当用户持续回访、主动询问新故事或额外结局，并且内容成本与使用需求得到验证后，再通过访谈和无支付承诺的价格敏感度测试评估商业化。

### 2026-07-18：首条垂直切片允许用户设定故事并使用本地 Mock 生成

- **时间**：2026-07-18
- **决定**：将四项轻量故事设定纳入首条垂直切片；生成阶段只调用本地 Mock，输出不超过 7 个节点的 Zod 合法分支故事。当前故事和节点保存在 LocalStorage。此决定调整了 `docs/PRODUCT.md` 中原有的“用户自行创作故事”非目标，但不引入自由创作器、多故事管理或外部 AI API。
- **原因**：完整走通“设定—生成—游玩—结局”更能展示 AI 产品的核心交互形态，也能在接入真实模型前验证数据契约、分支逻辑和手机端流程。Mock 可保持结果稳定，并将技术风险限制在前端垂直切片内。
- **替代方案**：继续使用唯一预制故事、直接接入外部 AI API、使用数据库保存生成结果。预制故事无法验证创作入口；直接接入模型会提前引入成本和不确定性；数据库对单设备单故事恢复没有必要。
- **后续验证方式**：邀请目标用户独立完成从表单到结局的流程，观察完成率、输入理解成本和分支差异感。当 Mock 流程稳定且真实生成质量成为主要风险时，再单独决策是否接入外部 AI API。

### 2026-07-19：使用可替换的服务端生成层接入 OpenAI

- **时间**：2026-07-19
- **决定**：新增统一异步 `StoryGenerator` 契约，由 Mock 和 OpenAI 两种实现共同遵守。生成请求统一进入 Next.js 服务端 API；默认模式为 Mock，只有显式设置 AI 模式且存在服务端 API Key 时才调用 OpenAI。OpenAI 默认模型为 `gpt-5.6-terra`，请求超时为 20 秒且不自动重试。此决定是对 2026-07-18“暂不接入外部 AI API”的重新评估。
- **原因**：首条 Mock 垂直切片已经验证输入、分支结构、游玩和恢复流程，下一项主要风险是模型能否稳定生成符合既有 Schema 的故事。统一接口可以在不改变前端流程和故事数据结构的前提下替换实现；服务端边界可避免 API Key 暴露，并集中处理超时、输出校验和错误分类。
- **替代方案**：继续只使用 Mock 无法验证真实生成质量；在浏览器直接调用模型会暴露密钥和供应商细节；一次接入多个供应商会增加适配与测试成本；AI 失败后静默返回 Mock 会让用户误判内容来源。因此本阶段只接入 OpenAI，失败时显示明确提示，Mock 作为可显式切换的开发、测试和运维兜底。
- **后续验证方式**：使用 `docs/EVALUATION.md` 对代表性输入和选择路径评分，同时记录超时、供应商错误和非法输出比例。当 OpenAI 的质量或成本不能达到验证目标，或确有第二供应商需求时，再评估模型、Prompt、重试策略或新的生成器实现；任何扩展都不得改变现有 `BranchStory` 契约。

### 2026-07-20：模型输出必须经过结构校验和故事图规则校验

- **时间**：2026-07-20
- **决定**：模型原始输出必须依次经过 JSON 解析、Zod 结构校验和独立故事图规则校验，全部通过后才可作为 AI 故事返回。结构校验负责字段、类型、必填内容和节点上限；故事图校验负责引用、可达性、结局数量、循环和两次选择路径等跨节点约束。
- **原因**：结构化输出能提高格式稳定性，但“JSON 可解析”和“字段符合 Schema”都不能证明故事图可以玩完。例如 `nextNodeId` 可能不存在、结局可能不可达或分支可能形成循环。分层校验能精确识别故障阶段，也让规则脱离 API 和 React 组件独立测试。
- **替代方案**：只依赖模型提示词无法形成确定性边界；把所有规则写进一个 Zod `superRefine` 会混淆结构与图规则；由前端边游玩边发现错误会让用户在主流程中断。三者均不采用。
- **对质量、延迟和成本的影响**：显著提高可玩性和错误可诊断性；本地校验只增加极小延迟且不产生模型费用；拒绝非法输出可能增加一次重试成本，但避免把坏故事交给用户。
- **后续验证方式**：持续记录 `INVALID_JSON`、`SCHEMA_VALIDATION_ERROR` 和 `STORY_GRAPH_ERROR` 的比例，并为每条新业务规则补充独立测试。当同类错误持续集中时，优先调整 Prompt 或结构定义，而不是放宽核心可玩性规则。

### 2026-07-20：自动重试最多一次

- **时间**：2026-07-20
- **决定**：一次生成最多发起两次模型请求，即首次失败后最多自动重试一次。只有超时、网络错误、可恢复的模型错误以及可重新生成的 JSON、Schema 或故事图错误允许重试；无效用户输入、缺少配置、模型拒绝、非临时 4xx 和未知错误不重试。两次请求共用不超过 15 秒的总等待预算。本决定替代 2026-07-19 记录中的“20 秒且不自动重试”。
- **原因**：短暂网络波动和模型偶发格式错误常可通过一次新生成恢复，但继续重试会快速增加用户等待、调用成本和重复故障概率，不符合三分钟轻体验的定位。
- **替代方案**：完全不重试会放大偶发失败；固定多次重试或无限重试会让延迟和费用不可控；所有错误无差别重试会重复发送确定无法恢复的请求。
- **对质量、延迟和成本的影响**：一次重试可提升偶发故障恢复率，也可能接近翻倍单次失败请求的模型成本；15 秒总预算限制最坏等待时间。最终故事仍须通过全部校验，因此重试不降低质量门槛。
- **后续验证方式**：在结构化日志中比较首次成功率、重试恢复率、P95 总耗时和每次成功生成的平均调用次数。若重试恢复率低或 15 秒等待明显损害完成率，重新评估错误白名单和每次请求预算。

### 2026-07-20：Mock 降级必须显式标记

- **时间**：2026-07-20
- **决定**：AI 在允许的重试后仍失败时，默认降级到 Mock；可通过 `STORY_GENERATOR_FALLBACK=none` 关闭。降级结果保存 `source=mock`、`fallbackUsed=true` 和失败类别，游玩页显示“AI 暂时不可用，已为你加载示例故事。”。不得将其表现为 AI 成功。
- **原因**：降级能保护完整游玩流程，但静默降级会误导用户，也会把 Mock 的稳定内容混入 AI 质量数据，导致产品团队高估模型可靠性和故事质量。
- **替代方案**：AI 失败后一律中断能保持来源纯粹，但演示和测试体验更脆弱；静默返回 Mock 体验最顺滑，却损害信任和指标真实性；在生成页弹窗要求用户选择会额外打断流程。
- **对质量、延迟和成本的影响**：在不增加额外模型调用的情况下提高端到端完成率；提示会暴露一次服务失败，但保护用户知情权。AI 内容质量评测只统计 `source=openai` 且 `fallbackUsed=false` 的样本；Mock 降级率单独作为可靠性指标，不计入 AI 质量分母。
- **后续验证方式**：同时观察 AI 首次成功率、重试恢复率、Mock 降级率、降级提示后的故事完成率和用户困惑反馈。如果降级率过高，优先修复 Prompt 或模型链路；如果提示显著打断体验，再测试更轻量的文案位置，但不取消来源标记。
