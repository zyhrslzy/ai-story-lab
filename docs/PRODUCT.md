# 产品定义

## 用户画像

首批用户是喜欢氛围感视觉、治愈系故事和 AIGC 内容的小红书用户。他们通常在通勤、睡前或情绪低落等碎片时间消费内容，希望快速进入一个有情绪张力的世界，但不愿承担完整游戏或长篇故事的时间成本。

这类用户重视：

- 视觉与氛围是否能迅速建立沉浸感
- 自己的选择是否会得到明确反馈
- 结局是否具有个人意味，值得保存或分享

第一版不以硬核游戏玩家、长篇网文用户或专业内容创作者为目标用户。

## 用户问题

现有短视频足够好看，但体验通常被动且转瞬即逝；完整游戏和长篇故事又需要较高的时间投入。用户缺少一种能在碎片时间内同时提供沉浸感、参与感和情绪回报的轻量体验。

第一版需要验证的核心问题是：

- 三分钟互动能否带来明确的情绪回报
- 两次选择能否让用户产生“这是我的故事”的感觉
- 专属结局是否值得用户保存或分享

## 一句话价值主张

用四项简单设定、两次关键选择和多个分支结局，为喜欢氛围感内容的用户提供一段三分钟内可以创作并玩完的个性化故事。

## 用户流程

1. 用户从小红书内容、作品集链接或二维码进入免登录手机网页。
2. 用户填写故事主题、主角名字、主角身份和故事风格。
3. 用户点击“生成故事”，本地 Mock 生成器返回通过 Zod 校验的分支故事。
4. 系统保存故事和起始节点到 LocalStorage，并进入游玩页。
5. 用户阅读当前剧情并完成第一次二选一。
6. 用户阅读选择后的剧情并完成第二次二选一。
7. 用户到达至少两个不同结局之一。
8. 用户可以刷新页面继续当前故事，或在结局点击“重新创作”返回表单。

从点击“开始故事”到看到结局卡，目标完成时间不超过三分钟。

## MVP 范围

- 一个包含四项设定的故事创作表单
- 一个不调用外部服务的本地 Mock 故事生成器
- 使用 Zod 校验输入、故事结构和持久化会话
- 每个故事总节点不超过 7 个
- 每个普通节点固定提供两个选择，结局节点不提供选择
- 每条路径经过两次选择到达结局，并至少包含两个不同类型的结局
- 使用 LocalStorage 保存当前故事和当前节点，支持刷新恢复
- 结局页提供“重新创作”并清除当前会话
- 手机优先的简洁界面，不投入复杂视觉效果

## 非目标

第一版明确不做：

- 自由对话和无限续写
- 同时保存或管理多个故事、多章节或复杂分支树
- 外部 AI API 和运行时模型调用
- 运行时实时生成图片或视频
- 登录、账户体系、个人主页和跨会话历史记录
- 自建数据库及需要持久化的用户状态
- 社区、评论、排行榜和好友系统
- 道具、积分、支付和其他商业化功能
- 配音、数字人或复杂动画
- 原生 App 和微信小程序

## 用户故事

- 作为一名从小红书进入的用户，我希望无需注册即可开始故事，以便立刻判断这个体验是否吸引我。
- 作为一名故事体验者，我希望填写少量设定即可生成故事，以便主角和主题与我有关。
- 作为一名只有几分钟空闲时间的用户，我希望故事能在三分钟内完成，以便在通勤或睡前获得一次完整体验。
- 作为一名参与故事的用户，我希望每次选择后都看到明确反馈，以便相信自己的选择影响了故事。
- 作为一名完成故事的用户，我希望获得与我的选择相关的专属结局，以便感到这段体验属于我。
- 作为一名喜欢表达自我的用户，我希望能保存或分享结局卡，以便向朋友展示我的结果。
- 作为一名对其他结果好奇的用户，我希望能从头重玩，以便探索另一条选择路线。

## 验收标准

### 免登录创作

- Given 用户通过外部链接或二维码首次进入产品
- When 用户填写四项有效设定并点击“生成故事”
- Then 系统不要求登录或注册
- And 本地 Mock 生成器返回通过 Zod Schema 校验的分支故事
- And 系统保存故事并进入游玩页面

### 选择与反馈

- Given 用户到达任一选择节点
- When 用户选择其中一个选项
- Then 系统记录该选择，并在下一段内容中显示与该选项对应的差异化反馈

### 结局到达

- Given 用户已经完成两次选择
- When 系统进入选择指向的结局节点
- Then 页面显示结局内容和结局类型
- And 页面不再显示故事选择
- And 页面显示“重新创作”按钮

### Schema 拒绝非法故事

- Given 故事超过 7 个节点、普通节点不是两个选择、选择引用不存在的节点、包含循环或少于两个不同结局
- When 系统使用 Zod Schema 解析故事
- Then 解析失败且非法故事不得进入游玩流程

### 刷新恢复

- Given 用户已经生成故事或完成至少一次选择
- When 用户刷新游玩页面
- Then 系统从 LocalStorage 恢复通过 Schema 校验的故事和当前节点

### 重玩

- Given 用户已经到达结局
- When 用户点击“重新创作”
- Then 系统清除 LocalStorage 中的当前故事并返回创作表单

## 初始指标

首轮验证至少邀请 30 名目标用户，观察以下指标：

| 指标 | 定义 | 初始目标 |
| --- | --- | --- |
| 故事完成率 | 点击“开始故事”后成功看到结局卡的用户数 ÷ 开始故事的用户数 | ≥ 60% |
| 结局卡分享意图率 | 看到结局卡后点击保存、系统分享或复制分享文案的用户数 ÷ 看到结局卡的用户数 | ≥ 15% |
| 主动重玩率 | 看到结局卡后点击“重玩”的用户数 ÷ 看到结局卡的用户数 | ≥ 20% |

结局卡分享意图率是第一版的核心价值指标。浏览量和停留时长仅作为辅助观察指标，不作为产品成立的主要依据。
